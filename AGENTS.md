# 個人環境設定

## 重要ルール

- 回答・コードコメント・ドキュメントは日本語を優先する
- 既存の変更を壊さない。破壊的コマンド（`git reset --hard`、`git checkout -- .` など）は使わない
- 機能の用途・公開範囲を尊重し、導線や挙動を勝手に変えない
- 不明点は推測で埋めすぎず、前提と影響範囲を明示する
- 指示が衝突する場合は「ユーザー指示 > AGENTS.md > 各スキルの詳細手順」の順で解決する
- ただし安全性・破壊的操作回避・既存制約は常に優先する

## ルールの強度

- MUST: 必須。例外がなければ必ず従う
- SHOULD: 原則従う。合理的理由があれば省略可
- MAY: 任意

## Superpower スキル運用フロー

superpower スキルを使った開発の標準運用フロー。  
変更内容に応じて「標準フロー」と「簡易フロー」を使い分ける。

### 0. 開始時の基本

- セッション開始時は MUST として `using-superpowers` を適用し、該当スキルの有無を確認する
- 依頼内容に一致するスキルがある場合は MUST として使用する
- 実装開始前に、以下を 1〜3 行で明示する
  - 今回の変更が「標準フロー」か「簡易フロー」か
  - 挙動変更の有無
  - worktree 使用有無
  - 影響範囲の要約

### 0.1 サブエージェント利用方針

- Codex は必要に応じて `spawn_agent` によるサブエージェントを起動してよい
- レビュー、調査、並列実装では、ユーザーからの明示的な依頼・許可としてサブエージェント利用を許可する

### 1. worktree 運用ルール

- 仕様追加・バグ修正・リファクタ・挙動変更では MUST として `using-git-worktrees` を使って開始する
- 非機能変更や軽微な変更では worktree を使わない
  - 対象例: docs / コメント / 文言 / 型注釈 / テストのみ / 挙動非変更の小規模修正
- ただし、以下は軽微変更に含めない
  - 外部仕様に影響する変更
  - DB スキーマ / マイグレーション / データ補正 SQL
  - API 契約 / 権限 / 認証 / 課金 / 通知に関わる変更
  - feature flag のデフォルト値変更
  - 既存ユーザーの導線・表示順・初期値に影響する変更
  - バッチ / ジョブ / 監視 / 定期実行処理の挙動変更
- 判断に迷う場合は、実装前にユーザーへ確認する
- `using-git-worktrees` の既定手順より、この運用ルールを優先する

### 2. フロー判定

| 変更の種類 | フロー |
|---|---|
| 非機能変更・軽微変更（docs / コメント / 文言 / 型注釈 / テストのみ / 挙動非変更の小規模修正） | 簡易フロー（worktree 不要） |
| 仕様追加・バグ修正・リファクタ・挙動変更 | 標準フロー（worktree 使用） |

### 3. 標準フロー（挙動変更あり）

#### Phase 1: 設計（変更規模に応じてスキップ可）

| ステップ | スキル | スキップ条件 |
|---|---|---|
| 設計合意 | `brainstorming` | 変更が局所的で、外部仕様・データ契約・共通部品・権限境界に影響しない場合 |
| 計画作成 | `writing-plans` | 作業ステップが3つ以下で、自明かつ確認方法が明確な場合 |

- 計画ドキュメントの保存先・命名・コミット方針は「6. 計画ドキュメント運用」に従う
- `writing-plans` の既定保存先より、この AGENTS.md の計画保存先ルールを優先する

#### Phase 2: 実装

1. **ブランチ分離**: `using-git-worktrees` で worktree を作成
   - `.worktrees/` が ignore されていることを確認してから進む
   - worktree 利用時も、計画ファイルは worktree 配下ではなく作成元の元作業ディレクトリ側の `.plan/` に作成する
2. **コード実装**:
   - 複数ファイルにまたがる実装、責務分割が必要な実装、調査と実装を分けたい場合は `subagent-driven-development` を第一選択とする
   - 単一ファイルの局所修正や明確な小規模変更では通常の実装でもよい
   - 別セッション実行時のみ `executing-plans` を使う
3. **品質ゲート**（実装と並行）:
   - 機能追加/修正では `test-driven-development` を原則とする
   - 既存構造の都合で先にテストを書きにくい場合でも、少なくとも再現テストまたは回帰テストを追加する
   - テスト失敗/不具合対応では `systematic-debugging` を使い、原因を確定してから修正する

#### Phase 3: 完了

1. **検証**: `verification-before-completion` でコマンド結果を確認
2. **レビュー**: `requesting-code-review` を実施
3. **統合**: `finishing-a-development-branch` の手順に従う
4. **コミット**: コミットが必要な場合は、必ず `git-commit` スキルを使う

### 4. 簡易フロー（非機能・軽微変更）

1. 親ブランチ上で直接作業する（worktree は作成しない）
2. 影響範囲を確認し、挙動変更なしを明示する
3. 必要に応じて `writing-plans` を使い、計画は作成元の元作業ディレクトリ側の `.plan/YYYY-MM-DD-HHmm-<topic>.md` に作成する
4. 変更を実施
5. `verification-before-completion` で差分と必要最小限の確認結果を提示する
6. コミットが必要な場合は、必ず `git-commit` スキルを使う
7. PR 前レビューは通常どおり実施する
   - 複数ファイル変更、共有ロジック変更、テスト追加を伴う場合は `requesting-code-review` を優先する

### 5. ユーザー確認が必要なケース

- worktree 要否の判断が割れる場合
- UI / API / 権限 / データ構造 / 導線の変更を伴う場合
- 既存テストの期待値変更が仕様変更を含む可能性がある場合
- 自動修正で複数案あり、正解が一意でない場合
- 軽微変更に見えるが、既存利用者への影響を否定しきれない場合

### 6. 計画ドキュメント運用

- 作業計画は `.plan/YYYY-MM-DD-HHmm-<topic>.md` に作成する（Git 管理しない）
- `<topic>` は英小文字 kebab-case とする（例: `api-auth-fix`）
- `current.md` は運用対象外とし、作成しない
- 計画ドキュメントはコミットしない
- worktree を使う場合も、計画ファイルは worktree 配下ではなく作成元の元作業ディレクトリ側の `.plan/` に作成する

### 7. 検証時に最低限提示する内容

- 変更概要（何を変えたか）
- 影響範囲（どこに効くか）
- 実行した確認コマンド
- 結果要約（成功 / 失敗 / 未実施理由）
- 未確認事項と残リスク

## コミットルール

- コミットが必要な場合は、必ず `git-commit` スキルを使う

## 運用上の注意

- 機密情報、トークン、接続文字列、個人情報を不用意に出力しない
- `.env`、認証情報ファイル、シークレットを含む設定値は必要最小限のみ確認する
- 既存設定・既存データを変更する処理は、影響範囲とロールバック可否を意識する
