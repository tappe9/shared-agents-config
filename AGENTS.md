# AGENTS.md

このファイルは、AI コーディングエージェント（ChatGPT / Copilot / Claude など）向けの共通運用ガイドです。  
チーム共通のルール・スキル運用はこのファイルを参照元としてください。

## 重要ルール

- 回答・コードコメント・ドキュメントは日本語を優先する
- 既存の変更を壊さない。破壊的コマンド（`git reset --hard` など）は使わない
- 機能の用途・公開範囲を尊重し、導線や挙動を勝手に変えない

## コミュニケーション / 作業スタイル

- 返答は簡潔に要点中心。必要な背景のみ補足する
- コードコメントは自明でない箇所に短く書く
- 自動生成・一括置換は必ず差分確認後に適用する
- 依頼がない限り既存の構成・命名・デザインシステムに従う

## Superpower スキル運用フロー

superpower スキルを使った開発の標準運用フロー。  
基本は「実務ハイブリッド」運用とし、挙動変更を伴う変更は標準フロー、非機能変更のみ簡易フローを許可する。

### 0. 開始時の基本

- セッション開始時はまず `using-superpowers` を適用し、該当スキルの有無を確認する
- 依頼内容に一致するスキルがある場合は必ず使用する
- 回答・コメント・ドキュメントは日本語を優先する

### 1. フロー判定

| 変更の種類 | フロー |
|---|---|
| 非機能変更（docs / コメント / 文言 / 型注釈など、挙動を変えない変更） | 簡易フロー |
| 仕様追加・バグ修正・リファクタ・挙動変更 | 標準フロー |

### 2. 標準フロー（挙動変更あり）

1. **`brainstorming`** — 設計合意
   - 目的・制約・成功条件を明確化し、設計合意を取る
2. **`writing-plans`** — 計画作成
   - 作業計画は各 worktree のローカルファイル `.plan/current.md` に作成する
   - 恒久的に残す価値がある計画のみ `docs/plans/YYYY-MM-DD-<topic>.md` に保存してコミットする
3. **`using-git-worktrees`** — ブランチ分離（コード変更時は原則必須）
   - 実装は worktree で開始する（docs のみ変更は除く）
   - `.worktrees/` または `worktrees/` を使う場合は `git check-ignore` で ignore 状態を確認する
   - ignore されていない場合は `.gitignore` に追加してから進む
4. **実装実行**（デフォルト）
   - `subagent-driven-development` を第一選択とする
   - 別セッションで実行したい場合のみ `executing-plans` を使う
5. **品質ゲート**（実装中）
   - 機能追加/修正は `test-driven-development` を適用する
   - 不具合・テスト失敗時は `systematic-debugging` を先に適用し、原因確定前に修正しない
6. **`verification-before-completion`** — 完了前検証
   - 成功主張前にコマンド結果で確認する
7. **レビューと統合**
   - `subagent-driven-development` を使う場合は各タスク後に `requesting-code-review` を必須とする
   - major feature 完了時と PR 前は `requesting-code-review` を必須とする
   - 最終統合は `finishing-a-development-branch` の手順に従う

### 3. 簡易フロー（非機能変更のみ）

1. 影響範囲を確認（挙動変更なしを明示）
2. 変更を実施
3. `verification-before-completion` で差分と必要最小限の確認結果を提示
4. PR 前レビューは通常どおり実施する

### 4. 運用上の注意

- 指示が衝突する場合は「ユーザー指示 > AGENTS.md > 各スキルの詳細手順」の順で解決する
- ただし安全性・破壊的操作回避・既存制約は常に優先する

### 5. 計画ドキュメント運用

- 作業計画は各 worktree のローカルファイル `.plan/current.md` に作成する
- `.plan/` は Git 管理しない（必ず ignore する）
- 恒久的に残す価値がある計画のみ `docs/plans/YYYY-MM-DD-<topic>.md` に保存してコミットする
- 恒久化の基準は「将来の実装判断で再利用する設計判断・制約・トレードオフを含むこと」

## コミットルール

- コミットが必要な場合は、必ず `git-commit` スキルを使う
